# Reno Compass — Consolidated Change Log & Architectural Decisions

This document logs all design plans, code revisions, architectural choices, and verification results implemented during the current iteration.

---

## 1. UI Redesign & Key Event Handling

### Send Button & Text Area Restructuring
* **Decisions**: Relocated the **Send** button to its own row (`.chat-actions-row`) directly beneath the `#chatInput` text area. This prevents the button from stretching vertically and enhances visual layout balance. Increased textarea `min-height` to `100px`.
* **Keyboard Triggers**: Implemented event handler logic to distinguish key entries:
  * **Enter**: Inserts a standard newline (`\n`) for comfortable multi-paragraph message writing.
  * **Shift + Enter**: Triggers chat submission.

### Header Cleanup & Responsiveness
* **Decisions**: Removed the manual "Approve & Advance" button and the redundant "Reset" button from the header panel. Relocated the remaining session management buttons to the header bar.
* **Responsiveness Breakpoints**: Appended mobile-first media queries to handle scaling on devices under 1100px and 768px, ensuring that the step tracker handles overflow by scrolling horizontally rather than squishing elements.

---

## 2. Flat Client Payloads & Response Filtering

### Flat JSON API Response Structures
* **Decisions**: Removed nested `"dossier"` wraps. Response payloads from routes like `/api/session/new`, `/api/session/load`, `/api/chat`, and `/api/session/restore-pdf` return flat, safe state representations:
  ```json
  {
    "current_stage": "scope",
    "conversation": [...]
  }
  ```
  This simplifies frontend parsing in `app.js` and prevents raw database structure exposure.

---

## 3. Extraction & Validation Updates

### Stated Preferences Filtering
* **Decisions**: Replaced unified prompt extraction with role-based log segregation. Stated properties (like targets, zipcodes, and constraints) are compiled strictly from user-only turns (`role == "user"`). Explanatory ballpark ranges generated by the agent (`role == "agent"`) are excluded.

### Numerical Defaults and Boundary Validation
* **Decisions**: Replaced placeholder numbers with explicit invalid defaults (`-1.0` or `"-1"`) to prevent the system from skipping gate checks.
* **Target-to-Ceiling Fallback**: If a valid `budget_target` is provided by the customer but the `budget_ceiling` is uninitialized, the system maps `budget_ceiling = budget_target`.
* **Stage Gate Evaluator**: Refactored the scope stage gate to block advancement if `budget_target <= 0.0` or equal to `-1.0`.

---

## 4. Inline Auto-Advancement Interceptor

### Conversational Stage Transitions
* **Decisions**: Automated stage advancement to streamline progression.
* **Mechanism**: Added the `[APPROVE_STAGE_TRANSITION]` directive to system behavioral rules. The server intercepts this tag inside `/api/chat`, strips it from the response text, updates the database state (`user_final_verdict = True`), advances the pipeline stage, and immediately resolves and seeds the next stage's starting questions in a single response turn.

---

## 5. Client Grounding Tool Configuration Bifurcation

### Google AI Studio & Vertex AI Tool Layouts
* **Decisions**: Resolved tool validation errors caused by API differences between Google AI Studio and Vertex AI.
* **Resolution**:
  * **Google AI Studio SDK (`google.generativeai`)**: Configured to pass the string `"google_search_retrieval"` (or dictionary `{"google_search_retrieval": {}}`). It does not support `"google_search"`.
  * **Vertex AI SDK (`vertexai`)**: Configured to load grounding via `Tool.from_dict({"google_search": {}})` since `google_search_retrieval` is deprecated on Vertex AI's gRPC endpoints.

---

## 6. Storage Resiliency Fallback

### GCS NotFound Resiliency
* **Decisions**: Fixed a bug where a GCS `NotFound` exception bypassed the local storage disk caching fallback.
* **Resolution**: Updated `read_session()` in `src/data/storage.py` to check the local disk fallback cache on `NotFound` exceptions before returning `None`, ensuring read continuity when a GCS bucket is offline or unprovisioned.

---

## 7. Mock Schema Alignments

### Mock Domain Schemas
* **Decisions**: Fixed schema mismatches in mock extraction logic that triggered validation failures during stage transitions.
* **Resolution**: Updated all mock data instantiations in `_run_mock_extraction()` for the `design`, `safety_permit`, `materials`, `contractor_validation`, and `diy_planning` stages to match current Pydantic models (`DesignOption`, `DiyProcedure`, `MaterialLineItem`, `MaterialTotal`, etc.).

## 8. Live Evaluation & Integration Test Suite

### Quality Gate
* **Decisions**: Created a dedicated evaluations suite under `tests/evals/` containing 6 test cases to verify the exact scenarios of our resolved issues.
* **Coverage**:
  1. `test_tool_compilation.py`: Verifies Google AI Studio and Vertex AI SDK tool list layouts.
  2. `test_payload_exposure.py`: Asserts flat client payloads and lack of raw dossier leakage.
  3. `test_extraction_accuracy.py`: Validates role-segregated parameter parsing.
  4. `test_auto_advance_transition.py`: Tests transition token stripping and stage advance triggers.
  5. `test_storage_resiliency.py`: Tests GCS-to-local storage write/read fallbacks.
